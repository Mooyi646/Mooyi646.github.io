0 概述

为什么使用Helm?

K8s能够很好地组织和编排容器，但是缺少一个更高层次的应用打包工具。

组织应用的服务多时，k8s对其组织和管理会出现的问题：

  • 很难管理、编辑和维护大量服务。每个服务都有若干配置，缺乏更高层次的工具将这些配置组织起来
  • 不容易将这些服务作为一个整体统一发布。缺少工具定义应用与服务、服务与服务之间的依赖关系
  • 不能高效地共享和重用服务，不支持参数化配置和多环境配置
  • 不支持应用级别地版本管理。通过kubectl rollout undo只能进行单个Deployment的回滚，不支持整个应用的回滚
  • 不支持对部署的应用状态进行验证

而通过Helm就可以解决以上问题。

Helm作用：通过合并配置文件到一个可重用的包，从而可以自动创建、打包、配置、部署k8s应用。简单来说，Helm也就是一个Package manager。

概念介绍：

Chart：一个描述K8s相关资源的文件集合。

Release：一个运行的chart实例。

Helm repositories：即存放helm charts的库。 Artifact Hub: 类似于docker hub

Helm的适用场景：

适用场景：

  • 有多个需被统一管理和部署的k8s应用
  • K8s应用经常更新与部署
  • 多团队多用户合作开发部署k8s应用

2 Values

2.1 Values对象

作用：提供传递值到chart的方法。

内容来源：

  • values.yaml文件

  • 子chart不可访问父chart的值，但是可以被覆盖

  • 使用-f参数传递到helm install/helm upgrade的values文件

  • 使用--set传递的单个参数

    顺序：默认使用values.yaml，可以被父chart的values.yaml覆盖，继而被用户提供values文件覆盖， 最后会被--set参数覆盖，优先级为values.yaml最低，--set参数最高。

即Helm相当于将config value与具体的资源的yaml解耦，即将具体的配置字段即其值提取出来成一个values.yaml文件。

在资源具体的yaml文件中，字段值不直接配置，而是使用占位符代替。

针对不同的运行环境，无需更改具体的yaml配置文件，而是更改values.yaml，类似于抽象出接口提供修改，而不是每次都到具体文件中修改。

2.2全局值

使用完全一样的名字在所有的chart及子chart中都能访问的值。全局变量需要显式声明。不能将现有的非全局值作为全局值使用。

可以在values.yaml中设置global字段来设置全局值

3.1.4 空白控制

错误的空白/空格会导致错误的YAML，缩进需正确。

模板引擎运行时，会移除{{`和`}}里的内容，但是所留下的空白保持。

{{-`(包括添加的横杠和空格)表示向左删除空白， 而` -}}表示右边的空格应该被去掉。

要确保-和其他命令之间有一个空格。

一定注意空格就是换行

有时使用indent更好，如：{{ indent 2 "mug:true" }}

4 .helmignore文件

指定不想包含在helm chart中的文件，如果文件存在，helm package命令在打包时会忽略所有在其中匹配的文件

支持unix shell的全局匹配、相对路径匹配、反向匹配，每行只考虑一种模式

5 Chart Hook

作用：允许开发者在发布生命周期的某些点进行干预。

Hook的工作方式与常规模板类似，但是因为Helm对其不同的使用方式，会有一些特殊的注释。

5.1 可用的Hook

  • pre-install：模板渲染之后，k8s资源创建之前执行
  • post-install：所有资源加载到k8s之后执行
  • pre-delete：在K8s删除之前，执行删除请求
  • post-delete：在所有的版本资源删除之后执行删除请求
  • pre-upgrade： 在模板渲染之后，资源更新之前执行一个升级请求
  • post-upgrade：所有资源升级之后执行一个升级请求
  • pre-rollback：在模板渲染之后，资源回滚之前，执行一个回滚请求
  • post-rollback： 在所有资源被修改之后执行一个回滚请求
  • test：调用helm test子命令执行

5.2 钩子和发布生命周期

如为install定义的两个钩子：pre-install, post-install，在chart开发时两个钩子都执行，则整个install周期会变为：

  1. 用户执行 helm install foo
  2. Helm库调用安装API
  3. 在 crds/目录中的CRD会被安装
  4. 在一些验证之后，库会渲染foo模板
  1. 库准备执行pre-install钩子(将hook资源加载到Kubernetes中)
  2. 库按照权重对钩子排序(默认将权重指定为0)，然后在资源种类排序，最后按名称正序排列。
  3. 库先加载最小权重的钩子(从负到正)
  4. 库会等到钩子是 “Ready”状态(CRD除外)
  1. 库将生成的资源加载到Kubernetes中。注意如果设置了--wait参数，库会等所有资源是ready状态， 且所有资源准备就绪后才会执行post-install钩子。
  1. 库执行post-install钩子(加载钩子资源)。
  2. 库会等到钩子是”Ready”状态
  1. 库会返回发布对象(和其他数据)给客户端
  2. 客户端退出

什么叫等到钩子是Ready状态？

  • 资源是Job或Pod类型，Helm会等到直到其成功运行完成。若钩子失败，发布就会失败。这是一个阻塞操作，所以Helm客户端会在这个任务执行时暂停
  • 其他种类资源，一旦k8s将资源标记为已加载，资源会被认为是Ready。

当一个钩子中声明了很多资源时， 资源会串行执行。如果有钩子权重，会按照权重顺序执行。

钩子资源不使用对应版本管理

钩子创建的资源无法作为发布的一部分进行跟踪和管理。

一旦Helm验证钩子达到Ready状态，将不再使用钩子资源。

在钩子中创建了资源，不能依靠helm uninstall去删除资源。要删除这些资源，要么在钩子模板文件中 添加一个自定义的helm.sh/hook-delete-policy 注释，要么设置任务资源的生存时间（TTL）字段。

helm.sh/hook-delete-policy 注释值：

  • before-hook-creation：新钩子启动前删除之前的资源（默认）
  • hook-succeeded：钩子成功执行后删除资源
  • hook-failed：如果钩子执行失败，删除资源

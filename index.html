  • Cloneable：表明其具有拷贝能力，可以进行深拷贝或浅拷贝

    • 浅拷贝：新创建一个对象，该对象有着原始对象属性值的一份精确拷贝。即原始对象属性为基本类型，则拷贝基本属性的值，若其为引用类型，拷贝其内存空间地址（即不会新创建这个引用类型）
    • 深拷贝：将对象从内存中完整拷贝出来，即会在堆中创建新对象，若原始对象属性为引用类型，则会重新创建这个引用类型。（即不与原始对象共享内存）

  • Serializable：表明可以进行序列化操作。即将对象转化为字节流进行持久化存储或网络传输。

JDK 1.8之前底层数据结构分析

此时底层数据结构为：数组和链表结合使用，即链表散列

数据存储流程：

  1. HashMap通过key的hashCode经过扰动函数处理后得到hash值

  2. 然后通过(n - 1) & hash判断当前元素存放的位置（n为数组长度）

    (n - 1) & hash即为hash % (n)（当n为$2^x$时）

  3. 若当前位置存在元素，则判断该元素与要存入元素的hash值以及key是否相同

  4. 相同直接覆盖；不同则通过拉链法解决冲突

拉链法

数组与链表结合，即创建一个链表数组。遇到哈希冲突时，将冲突值加入链表即可。

JDK1.8之后底层数据结构分析

在解决哈希冲突时有了较大变化

在拉链法的基础上：

  1. 当链表长度大于阈值（默认为8），会首先调用treeifyBin()决定是否转换为红黑树

  2. 当hashMap数组长度>=64时，转换为红黑树以减少搜索时间

    红黑树节点数小于6时又会调整为链表

  3. 数组长度未达到64时，执行resize()方法对数组扩容。

源码分析

属性列表

  • loadFactor负载因子

    控制数组存放数据的疏密程度。

    • 越趋近于1，数组中存放的数据(entry)越多，也就越密，也就是会让链表长度增加（即扩容在满了才会发生）；

    • 越趋近于0，数组中存放的数据越少，也就越稀疏。

    大小控制：loadFactor 太大导致查找元素效率低，太小导致数组的利用率低，存放的数据会很分散。loadFactor 的默认值为0.75f是官方给出的一个比较好的临界值。

    • why 0.75

      $阈值(threshold) = 负载因子(loadFactor) * 容量(capacity)$

      元素个数超过threshold就会进行扩容

      由hashMap的机制，容量一直为2的幂。所以为了保证$负载因子(loadFactor) * 容量(capacity)$的结果为整数，loadFactor为0.75比较合理，因为其与2的幂的乘积都为整数。

resize()扩容

进行扩容，会伴随一次重新hash分配，并会遍历hash表中的所有元素，非常耗时。

实际上是将table初始化和table扩容进行了整合，底层行为都是给table赋值一个新的数组。

每次扩容都是扩大为原始大小的2倍。

为什么HashMap长度为2的幂

为存取高效、减少碰撞，需将数据尽量分配均匀。

Hash 值的范围值-2147483648 到 2147483647，前后加起来大概 40 亿的映射空间，只要哈希函数映射得比较均匀松散，一般应用是很难出现碰撞的。但问题是一个 40 亿长度的数组，内存是放不下的。所以这个散列值是不能直接拿来用的。

用之前还要先做对数组的长度取模运算，得到的余数才能用来要存放的位置也就是对应的数组下标。

我们首先可能会想到采用%取余的操作来实现。但是，重点来了：“ 取余(%)操作中如果除数是 2 的幂次则等价于与其除数减一的与(&)操作（也就是说 hash%length==hash&(length-1)的前提是 length 是 2 的 n 次方；）。” 并且采用二进制位操作 &，相对于%能够提高运算效率，这就解释了 HashMap 的长度为什么是 2 的幂次方。

其中：

  • type：即为查询类型，即在表中找到所需行的方式。有以下几种类型（从上到下，性能越来越好）：

    • ALL：全表扫描，即遍历全表以找到匹配的行。即直接遍历整个聚簇索引。

    • index：全索引扫描，通过遍历所有索引来查找匹配的行.

      直接在某个索引树上做条件判断，不需要回表。

    • range：索引范围扫描，常见于<, <=, >,>= between等操作符.

    • ref：普通的二级索引等值查询。

      可能会查到多个匹配值，性能比const差一点。

      由于null不能用const，所以不管是唯一索引还是二级索引值为null，都最多用到ref.

    • eq_ref：多表连接查询时，被驱动表通过主键或唯一索引键进行等值查询

    • const：使用主键或唯一索引等值查询来定位一条数据。因为索引的B+树都是矮胖的，定位一条唯一索引的记录，速度非常快，所以将该种查询定义为常数级，即为const.

      唯一索引中允许多个null存在，所以对其进行null值查询，无法使用const

    • system：是 const 类型的特例, 当查询的表只有一条数据的情况下使用 system.

    • NULL：不用访问表或索引就直接能得到结果

  • key：执行本条SQL时，是否用到索引。

  • Extra：额外的可选信息。可以通过其判断当前SQL的执行效率。主要有：

    • Using index：使用索引覆盖
    • Using index condition：使用索引下推
    • Using filesort：使用临时空间排序。

索引种类

目前MySQL提供了以下四种索引：

  1. B-Tree索引：最常见，大部分存储引擎都支持
  2. Hash索引：只有MEMORY引擎支持，使用场景简单
  3. R-Tree索引：又称空间索引，是MyISAM引擎的一个特殊索引类型，主要用于地理空间数据类型，通常使用较少。
  4. Full-text索引：又称全文索引，也是MyISAM引擎的一个特殊索引类型，主要用于全文检索，innoDB从mysql5.6开始支持全文索引。

索引结构

问题

Q1：索引过大无法加载到内存中怎么解决？

A1：分块读取。

同时需要提高IO效率，在软件层面可以使用辅助手段提高。

操作系统相关：

  1. 局部性原理：

    • 时间局部性：之前访问过的数据很可能再次被访问，至少比陌生数据概率大很多
    • 空间局部性：数据与程序都有聚集成群的倾向

  2. 磁盘预读：

    内存与磁盘交互的最小逻辑单位：页

    其大小一般为4KB或8KB，由操作系统决定。在内存进行数据读取时，一般会读取页的整数倍。

InnoDB引擎在进行数据加载时读取的就是16KB的数据，即为分块读取的集中体现。

Q2：why B+树？

索引存储的数据格式：key-value形式。

能够存储key-value格式的数据结构有很多，如：hashmap，树，为什么最终选用B+树做索引？

首先，why not hashmap？

  1. 利用hash存储需要将所有数据添加到内存，耗费内存空间
  2. 不合适的hash算法会导致hash冲突，以及会导致数据散列不均匀，空间浪费
  3. 哈希表存的key-value是无序的，进行范围查找时，需要遍历其中存储的所有数据，代价大。

InnoDB自适应Hash。

什么是自适应Hash？

InnoDB引擎会监控对表上二级索引的查找，如果发现某二级索引被频繁访问，二级索引成为热点数据，建立hash索引可以带来速度上的提升，则：

  1. 自适应hash索引功能被打开
  2. 经常访问的二级索引数据会被自动生成到hash索引中去（最近连续被访问三次的数据）。

自适应hash索引通过缓冲池的b+树来构造，所以建立速度很快。

特点

  1. 无序，没有树高
  2. 降低对二级索引树的频繁资源访问
  3. 自适应

缺点

  1. 会占用innodb buffer pool
  2. 只适合搜索等值的查询，范围查找无法使用
  3. MYSQL自动管理，无法人为干预

其次，why not 二叉树、平衡二叉搜索树、AVL树、红黑树？

以上四种数据结构的共同点为：都为二叉树。即其每个分支上最多两个节点。

当向其中插入大量数据时，无论其是否有序是否平衡，其树高度都会很大，导致在查找时，依然会存在多次比较操作，每次比较都相当于一次磁盘IO操作，会增加IO次数，影响效率

为什么每次比较都相当于一次磁盘IO?

因为每个节点占用一个磁盘块，在比较时会将其中数据读取到内存中。

则为了解决以上问题，可以从两个维度思考：

  1. 增加单个节点的范围：即单个节点的存值个数增加
  2. 增加节点的孩子个数

因此，B树与B+树应运而生。

B树

其为一棵平衡搜索多叉树。多叉，即为其孩子节点个数，也叫阶数，当我们描述一棵B或B+树时，需要指定其阶数，一般用m表示。

定义：

  • 每个节点都有key-value
    • 根节点至少有1对
    • 非根节点至少有$\lceil m/2 \rceil$对
    • 每个节点最多有m-1对
  • 所有叶子节点位于同一层
  • 除叶子节点外，所有节点的孩子数量为其当前的key-value对数 + 1
  • 每个节点中的key都按照排序规则从小到大排列，该节点左子树中所有key都小于它，右子树都大于它【开区间】

查询过程为：如需查询key = 28的数据

  1. 由根节点找到磁盘块1，读入到内存中
  2. 比较得出28在区间16 ~ 34，找到磁盘块1中的指针p2
  3. 由p2找到磁盘块3，读到内存中
  4. 比较得出28在区间25 ~ 31，找到其中的指针p2
  5. 由p2找到磁盘块8，读到内存中
  6. 比较得出key=28的key-value，结束

优点：降低了树高，虽然每次读进内存中的数据多，但是读取的次数少

缺点：每个节点都有key-value，磁盘块/页的存储空间是有限的，所以当value很大时，能存储的key的数量减少。

如何解决以上问题：

在非叶子节点的磁盘页中不存value，只存储key，具体的key-value全部存储到叶子节点中。即B+树的方案。

B+树

在B树的基础上进行优化，不同点为：

  • 非叶子节点只存储Key，叶子节点存储key-value
  • 每个叶子节点都有相邻叶子节点的指针
  • 子节点为左闭右开
  • 除叶子节点外，所有节点孩子数量等于当前的key-value数量

查询数据：

存在两种搜索方式：

  1. B树的搜索方式：$O(logN)$
  2. 链表搜索：叶子节点构成链表形成搜索，从最左下的叶子节点开始顺序搜索。

优点：

  1. 非叶子节点只存储key，所以相比B树，能存储更多的Key，因此能更大地降低树高与将范围分成更多的区间，数据检索更快。
  2. 叶子节点都有相邻节点的指针，符合磁盘预读的特性，范围查询性能更快
  3. 一般三到四层的B+树足以支撑千万级别的数据存储，更大则需分表

B+树所能存储的数据量计算：

  • 非叶子节点指针数：

    InnoDB一个页的大小为16K，假设索引key的bigint占8个字节，指针占字节数为6，8 + 6 = 14，则一个page所能存放的key和指针个数为：16k / 14 ，约等于1170

  • 叶子节点记录数：

    常规互联网单条行记录大小约为1k，则一个page能存储16k / 1k = 16条

    • 如果为2层，则能存储记录数为：1170 * 16 = 18720
    • 如果为3层，则能存储：1170 * 1170 * 16 约为2190w

索引设计

由上文分析，B+树非叶子节点的空间占用时，指针占用的为固定空间大小6个字节，所以影响其内存占用的因素为key

  • key为int：都占用4个字节

  • key为char：根据指定长度占用（英文字符一个长度一个字节，中文：utf8 - 3个, gbk - 2个）

    char(n) / varchar(n)：n表示字符数

    char(n) vs. varchar(n)

    • char(n)：
      • 不管实际value的长度，都会占用n个字符的空间。【不够长则填充空格，固定长度】
      • 存储时会自动截断尾部空格
      • 上限255字节
      • 不指定n时默认为1
    • varchar(n)：
      • 占用value的实际空间 + 1，且该值 <= n。【可变长度】
      • 不会截断尾部空格
      • 上限65535字节【2^16】
      • 不指定n时报错

    空间占用超过n后，字符串会被截断

    varchar vs. text

    • text：用于保存文本类型的字符串，有四种类型：
    文本字符串类型 特点 长度 长度范围 占用的存储空间
    TINYTEXT 小文本、可变长度 L 0 <= L <= 255 L + 2 个字节
    TEXT 文本、可变长度 L 0 <= L <= 65535 L + 2 个字节
    MEDIUMTEXT 中等文本、可变长度 L 0 <= L <= 16777215 L + 3 个字节
    LONGTEXT 大文本、可变长度 L 0 <= L<= 4294967295（相当于4GB） L + 4 个字节

    由于实际存储的长度不确定，MySQL 不允许 TEXT 类型的字段做主键

    varchar转换为text

    1. 大于varchar（255）变为 tinytext
    2. 大于varchar（500）变为 text
    3. 大于varchar（20000）变为 mediumtext

    为什么经常使用varchar(255)？

    MYSQL5.6及以前，InnoDB支持的前缀索引最大为767个字节，采用utf8编码时， $\lfloor 767 / 3 \rfloor = 255$

    255刚好为能建立前缀索引的最大值，不是最优，但是能减少出错

目标：让key占用尽可能少的空间

前缀索引

索引的选择性Index Selectivity = 基数 / 数据表的总记录数【即为唯一索引值占比】

基数为不重复索引值的数量，即唯一索引值数量

MYSQL中使用HyperLogLog做基数统计，这是一个预估值。

索引的选择性越高，查询效率越高，但是所占用的空间也就越多。

前缀索引定义

选择目标列的部分字符作为索引，从而节约空间，提高索引效率，但是会降低索引的选择性。【性能与空间之间寻找平衡】

大部分存储引擎都支持前缀索引，目前只有字符类型或二进制类型的字段可以建立前缀索引，如：CHAR/VARCHAR、TEXT/BLOB、BINARY/VARBINARY

  • 字符类型基于前缀字符长度
  • 二进制类型基于字节大小
  • TEXT/ BLOB类型只支持前缀索引，不支持整个字段建索引。

怎么选定合适的前缀？即如何确定合适的前缀长度？

  1. 统计指定字段出现重复值最多的前10个以及其重复出现次数

  2. 统计该字段前3个字符出现重复值最多的前10个以及其重复出现次数

    left(str , length)返回指定str从左边返回length长度的字符串

  3. 观察第1次与第2次统计结果差距大不大，如果大，则向右扩1个字符，在比较前3个和前4个字符的统计结果【如果仍然差距大，继续扩展重复】

  4. 知道结果差不多，则当时的前n个字符可作为前缀索引的最终长度，以此建立前缀索引

优点：索引更小，节约索引空间，提高索引效率

缺点：

  • 降低索引的选择性
  • 无法使用前缀索引做groupby和orderby，也无法做覆盖扫描

聚簇 / 非聚簇索引

并不是一种单独的具体索引类型，而是一种数据存储方式，具体依赖于其实现方式。

聚簇索引

即Clustered Index。

定义：按表的主键构造一棵B+树，叶子节点存放表的行记录数据，即将表的索引与数据存储在一起，找到索引的同时也找到了数据。

按照以上定义，很显然一张表只能有一个聚簇索引。

InnoDB中存储主键选用的就是聚簇索引的方式，如果未定义主键，则会选择非空唯一性(not null unique)字段，若不存在，则会隐式定义一个长整型，占6个字节的rowid作为主键来作为聚簇索引。

非聚簇索引

Non-Clustered Index，也叫辅助索引或二级索引。

非聚簇索引为在聚簇索引上创建的索引，即将数据与索引分开存储。

常见非聚簇索引：

  • 复合索引
  • 前缀索引
  • 唯一性索引
  • MyISAM中的主键做因

同样由B+树构成，但是树的叶子节点不再存储表的行记录数据，而是主键或指定索引字段

  • InnoDB中，存储的未主键
  • MyISAM中，主键索引存的是主键。非主键索引存的是指定索引字段。

如：存在表user，id为主键，name字段有索引，除此之外还有一个字段为company

聚簇索引优缺点：

优点：

  1. 查找速度快于非聚簇索引，因为在InnoDB中无需二次查询，在MyISAM中无需寻址
  2. 聚簇索引行数据在叶子节点中，同一个数据页可能有多条行数据，访问同一数据页的不同行数据是，其已被加载到内存中，再次访问可以直接在内存中访问，减少了IO
  3. InnoDB中使用主键值做非聚簇索引的value而非地址，是为了保证B+树分裂时，出现行移动的情况下，主键索引的B+树的变化不会影响非主键索引。
  4. 适用于排序场合、选取一定范围数据的场景

缺点：

  1. 一张表中只能有一个聚簇索引，可以有多个非聚簇索引

  2. 维护聚簇索引很昂贵，因为行移动会造成碎片

  3. 若使用UUID(通用唯一标识码)做主键，其本身特性会导致数据稀疏，则会出现使用非聚簇索引查询速度慢于全字典扫描，因此建议用自增主键做聚簇索引。

    为什么？

    因为聚簇索引的数据的物理存放顺序与索引顺序是一致的，只要索引相邻，则对应数据也存放在相邻磁盘上。

    如主键不是自增id，则不可能顺序插入，存储引擎会不断调整数据的物理地址，会不断分页。

    若主键为自增id，则只需一页一页写入数据，索引结构也相对紧凑，磁盘碎片少，效率更高。

如果涉及到大量数据的排序、全表扫描、count之类的操作的话，建议使用MyISAM引擎。因为所有占用空间小，这些操作是需要在内存中完成的。

什么条件查询才能利用联合索引呢？

由以上B+树，b字段只有在a字段固定时才有序，所以：

  1. where b = 2：无法使用联合索引，索引整体按照a字段排序，无法单独确定b
  2. where a = 2 and b = 2：可以使用
  3. where a > 2 and b = 2：a使用，b未使用。在a字段的固定范围中，b依然整体是无序状态

由以上可以得出最左匹配定义：查询条件中，以联合索引最左边定义的字段为起点，且所有字段和顺序都与联合索引中一致，构成的查询条件能够利用联合索引，查询条件中出现范围查询、模糊查询、排序则需具体分析。

注：查询条件没有按联合索引的顺序排列时：同级查询会被MYSQL中的查询优化器自动优化查询顺序，即尽量会满足按照联合索引顺序排列。MySQL的查询优化器无法对于group by，order by进行优化

索引下推

index condition pushdown，ICP，全称为索引条件下推优化。MySQL5.6推出，默认开启，用于优化查询。

使用索引下推的表示：explain 结果中的Extra值为Using index condition

原理

MySQL架构：

  • Client
  • Server
  • 存储引擎

未使用ICP时，MySQL如何查询？

  1. 存储引擎读取索引记录
  2. 根据索引中的主键值，定位并读取完整的行记录
  3. 存储引擎把记录交给Server层去检测该记录是否满足WHERE条件

使用ICP后，查询过程为：

  1. 存储引擎读取索引记录（不是完整的行记录）
  2. 判断WHERE条件部分是否能用索引中的列来做检查【即做一次筛选过滤】
    • 条件不满足则处理下一条索引记录
    • 条件满足，回表读取完整行记录
  3. 存储引擎把记录交给Server层，Server层检测该记录是否满足WHERE条件的其余部分

使用条件

  • 只能用于range, ref, eq_ref, ref_or_null访问方法

  • 只能用于InnoDB和MyISAM存储引擎及其分区表

  • 对InnoDB而言，ICP只适用于二级索引

    为什么？

    因为ICP的目的是减少回表次数，而对聚簇索引而言，其完整的行记录已经在缓存区，ICP无意义。

  • 引用了子查询的条件不能下推

  • 引用了存储函数的条件不能下推，因为存储引擎无法调用存储函数。

count()性能问题

count()作用

count()作用：统计符合查询条件的记录中，函数指定的参数不为 NULL 的记录有多少个。

  • count(name)：统计表中name不为空的记录条数
  • count(1)：在当前表中，1永远不为空，所以统计的为当前表的记录总数
  • count(*)：当前表中记录总数

count(primary key)执行过程

  1. Server层维护名为count的变量
  2. Server层循环向InnoDB中读取一条记录，id不为NULL，则count += 1
  3. 所有记录读取完毕，退出循环，将count变量值发送给Client

InnoDB遍历选择：

  • 无二级索引：遍历聚簇索引
  • 有二级索引：遍历二级索引【树更小】【优化器选择】

count(1)执行过程

InnoDB遍历选择：

  • 无二级索引：遍历聚簇索引，但是不会读取value
  • 有二级索引：遍历二级索引【树更小】【优化器选择】

count(*)执行过程

对于select *而言是读取所有记录，但是对于count不是

count(*)相当于count(0)，所以，其执行过程与count(1)相同

count(字段)执行过程

如count(name)，name不为索引

会采用全表扫描的方式，所以效率很低。

性能对比

count(1) = count(*) > count(primary key) > count(字段)

count(*)优化

数据量过大时，count(*)也很耗时，如何优化？

  1. 近似值：当无需精确值统计时，可以使用近似值，如使用show table status或explain来估算

  2. 额外表保存字段：如果是想精确的获取表的记录总数，我们可以将这个计数值保存到单独的一张计数表中。

    当我们在数据表插入一条记录的同时，将计数表中的计数字段 + 1。也就是说，在新增和删除操作时，都需要额外维护这个计数表。
